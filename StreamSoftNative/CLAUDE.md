# StreamSoft Native — правила проекта

Нативная версия StreamSoft (лежит внутри `softforstream/StreamSoftNative`; родительская
папка `softforstream` — эталон фич и протоколов на Python, но код оттуда сюда **не**
переносится и не шарится, это независимый подпроект). Цель: обычный пользователь качает
один инсталлятор, жмёт «Далее» пару раз — и получает работающий чат/алерты/оверлей, без
Python, без `pip install`, без ручного добавления Browser Source в OBS.

## 1. Главный архитектурный закон: C++ core, Python — только адаптер

- **Core** (эта папка, `core/`) пишется на C++ (C++20, CMake, MSVC — на этой машине уже
  стоит Visual Studio 2022 Build Tools с VC++ tools, используем его). Никаких Python-
  зависимостей в core в принципе — ни `python.exe` в PATH, ни встроенного интерпретатора.
- Python используется **только** там, где фича фундаментально завязана на Python-
  экосистему и переписать её на C++ нецелесообразно:
  - **TTS** — `edge-tts` (обёртка над недокументированным протоколом Microsoft Edge
    Read Aloud), нативного C++ эквивалента нет.
  - **RVC** — voice conversion, завязан на `torch`/CUDA, вся экосистема моделей на Python.
- Всё остальное — чтение чата Twitch IRC/YouTube API/Telegram API, локальный веб-сервер
  для оверлея, WebSocket-клиент к OBS (`obs-websocket`), трей-иконка, GUI настроек,
  модерация, чат-команды — **обязано** быть на C++. Если фичу можно сделать нативно —
  она делается нативно, это не обсуждается.
- Перед тем как писать что-то на Python, задай вопрос: "это правда невозможно на C++
  за разумное время, или просто так короче писать". Если второе — пиши на C++.

## 2. Модульность: core ничего не тянет за собой лишнего

Пользователь, которому нужны только алерты/чат-оверлей, не должен качать ни байта TTS
или RVC. Всё разбито на независимо устанавливаемые модули:

- **Core** (обязательный, лёгкий инсталлятор): чат-агрегация, Telegram-мост, веб-оверлей,
  авто-подключение к OBS, GUI настроек, трей. Никакого TTS, никакого RVC. Работает сразу
  после установки.
- **TTS-модуль** (опциональный): Python + `edge-tts` + минимальный embeddable-рантайм
  Python (свой, не системный — пользователь не должен ничего ставить сам). Качается
  отдельно, по кнопке.
- **RVC-модуль** (опциональный, тяжёлый — по образцу текущего `rvc_service/`, но там уже
  видели: 6.6GB с CUDA-сборкой torch). Качается отдельно, только для тех, у кого есть
  подходящая Nvidia-карта.

### "Check & Install" — обязательный паттерн для опциональных модулей

Кнопка «Установить» для TTS/RVC не должна просто бросать пользователя в скачивание
гигабайтов. Порядок действий:

1. Core **нативно** (без Python) проверяет требования модуля:
   - TTS: свободное место на диске, доступ в интернет — минимальные требования.
   - RVC: наличие Nvidia GPU и совместимость с CUDA (перечисление через DXGI/WMI,
     `Win32_VideoController` или NVML, без сторонних тяжёлых зависимостей), свободное
     место (≥8GB), опционально версия драйвера.
2. Если требования не выполняются — кнопка показывает **почему** конкретно нельзя
   («Нужна видеокарта Nvidia с поддержкой CUDA — у тебя интегрированная графика»), и не
   предлагает скачивание.
3. Если всё ок — кнопка "Установить" реально появляется, показывает размер закачки и
   качает пакет модуля (embeddable Python + adapter-код + зависимости, запакованные
   заранее — не `pip install` на лету у пользователя на глазах).

## 3. GUI настроек: Qt Quick (QML), не Widgets, не web

Решение прошло два пересмотра:
1. Первая попытка — WebView2 — отклонена пользователем: нужен по-настоящему нативный,
   "вау"-уровня полировки интерфейс, а не браузер в рамке.
2. Вторая попытка — **Qt Widgets + QSS** — тоже отклонена, уже после реализации и живой
   проверки: Widgets/QSS физически не тянут задачу. На практике всплыло два системных
   ограничения тулкита (не баги в нашем коде — ограничения самого Qt Widgets):
   - `QPushButton` без `setFlat(true)` продолжает рисовать нативный bevel style'а поверх
     любого QSS — борд-радиус игнорируется, кнопки остаются прямоугольными почти всегда.
   - Qt Style Sheets — не настоящий CSS3: `box-shadow`, `backdrop-filter`, `::after`,
     `var(--x)` не существуют в QSS-парсере и молча игнорируются. Реального блюра/стекла
     через QSS не сделать в принципе, только имитация цветом.
   Даже после точечных фиксов (см. §8, коммит с "black & white liquid glass") результат
   на живом скриншоте всё равно выглядел плоско и дёшево — не то, что просили ("вау").

- **Текущее и окончательное (пока не будет повода пересмотреть) решение — Qt Quick
  (QML) + Qt Quick Controls 2 (стиль `Basic`) + `QtQuick.Effects`** (`gui/`, тот же
  CMake-таргет/exe, полностью переписан). Причины:
  - QML рисует через сцен-граф, а не через `QStyle`/нативные виджеты — `Rectangle { radius: 20 }`
    просто рисует скруглённый прямоугольник, никакой борьбы с бевелом стиля.
  - `QtQuick.Effects` (`MultiEffect`, Qt 6.5+) даёт настоящий gaussian blur — можно
    сделать реальное frosted-glass стекло (`ShaderEffectSource` с фоном за карточкой,
    замазанным `MultiEffect.blurEnabled`), а не имитацию прозрачным цветом. Это
    физически недостижимо в Qt Widgets.
  - Живьём проверено на macOS через Homebrew Qt6 (см. §8) — блюр реально рендерится,
    кнопки/слайдеры/тумблеры/комбобоксы скругляются и стилизуются без единого обходного
    трюка вроде `setFlat`.
- Почему не WebView2 — как и раньше, не подходит под требование пользователя.
- Почему Qt ставится через `aqtinstall`, а не vcpkg — не изменилось, см. `docs/BUILD.md`.
- **Архитектурное разделение сохраняется**: `core/` (фоновый сервис — чат, оверлей,
  OBS-интеграция) остаётся БЕЗ Qt-зависимости, лёгкий, всегда работает в фоне без GUI.
  `gui/` — отдельный процесс, дёргает `core`'s REST API (`/api/settings` и т.д.) по
  localhost HTTP через `ApiClient` (`gui/api_client.hpp`, обёртка над
  `QNetworkAccessManager`, инвокабельна прямо из QML через JS-колбэки) — тот же REST-
  контракт, что был у Widgets-версии и у Python `settings_app.py`.
- В отличие от TTS/RVC-адаптеров (опциональные, качаются по кнопке) — GUI входит в
  основной core-инсталлятор по умолчанию: без него никто не сможет настроить Twitch/
  YouTube-подключение, это не "плюшка", а необходимость для обычного пользователя.

## 4. Контракт между core и адаптерами

Core не встраивает Python-интерпретатор и не импортирует Python-модули напрямую. Каждый
адаптер (TTS, RVC) — это отдельный локальный HTTP-микросервис на `127.0.0.1`, который
core запускает как подпроцесс и общается с ним по HTTP/JSON — **ровно так, как уже
работает `rvc_client.py`/`rvc_launcher.py` в `softforstream` для RVC**. Этот паттерн уже
проверен в существующем проекте — просто применяем его и к TTS, и делаем частью
архитектуры, а не частным случаем.

- Адаптер получает случайный свободный порт при старте, печатает его в stdout/файл, core
  его подхватывает.
- Если адаптер падает — core должен пережить это (см. `_supervise()` в
  `softforstream/main.py` как референс на паттерн изоляции сбоев) и просто отключить
  фичу (например, TTS замолкает, RVC-конвертация откатывается на обычный TTS-голос),
  а не падать целиком.
- Никакого IPC через shared memory/pipes с бинарной сериализацией — везде localhost
  HTTP + JSON, ради простоты отладки и потому что это уже отработанный в проекте подход.

## 5. Авто-интеграция с OBS

Плагин для OBS не пишем — используем встроенный (с OBS 28+) `obs-websocket` v5, порт
4455. Core сам:
- подключается к OBS по WebSocket,
- запрашивает `GetVideoSettings` (размер канвы),
- создаёт/находит browser source с URL на локальный `/events` и `/chat`, выставляет
  размер под канву, `Shutdown source when not visible = false`.

Если OBS не найден или websocket выключен — fallback: экспорт `.json` scene collection
для ручного импорта (File → Import), а не молчаливый отказ.

## 6. Сборка

- CMake + MSVC (Visual Studio 2022 Build Tools, уже установлены на машине разработки).
- Каждый модуль (`core/`, `adapters/*`) — независимый CMake-таргет/своя сборка.
- vcpkg для C++ зависимостей (пока не установлен на машине — ставим при первой реальной
  необходимости во внешней библиотеке, не раньше).

## 7. Отношение к `softforstream`

`softforstream` — работающий референс: протоколы Twitch IRC/EventSub/YouTube API/
Telegram, формат сообщений оверлея по WebSocket, JSON-схемы настроек. При портировании
фичи на C++ — сверяться с соответствующим Python-файлом там как со спецификацией
поведения, но не копировать код и не шарить директории/git-историю. Проекты полностью
раздельные.

## 8. Статус

- Тулчейн подтверждён: MSVC 19.44 (VS2022 Build Tools) + CMake + vcpkg (manifest-режим,
  `vcpkg.json`). Сборка: см. `docs/BUILD.md`.
- **Веб-сервер оверлея готов** (`core/include/overlay_server.hpp`, библиотека — Crow):
  `/`, `/chat`, `/events`, `/static/*` отдают статику из `core/web/` (пока — копия
  Python-версии `overlay/{index,chat,events}.html`, `style.css`, `app.js`, без изменений,
  т.к. формат WS-сообщений идентичен). `/ws` шлёт `config` + историю чата на коннект,
  рассылает `chat`/`event` всем клиентам. `/api/test-chat`, `/api/test-event` — временные
  ручки для проверки бродкаста без реальных чат-воркеров. Проверено вручную: HTTP-роуты
  (200/404), WS round-trip через `aiohttp`-клиент — совпадает с Python-референсом.
- **Чат-воркеры готовы**: `core/include/twitch_auth.hpp` (device-code flow + refresh +
  токен-кэш `twitch_token.json`, идентичен `softforstream/twitch_auth.py` по формату),
  `core/include/twitch_chat.hpp` (IRC поверх `asio::ssl`, TLS-хендшейк с проверкой
  сертификата и SNI), `core/include/youtube_chat.hpp` (HTTPS-поллинг через cpp-httplib).
  `core/include/env_config.hpp` — минимальный `.env`-лоадер (`TWITCH_CLIENT_ID`,
  `TWITCH_CHANNEL`, `YOUTUBE_API_KEY`, `YOUTUBE_VIDEO_ID`). Оба воркера запускаются из
  `main.cpp` в отдельных потоках, каждый обёрнут в `supervise()` — тот же паттерн
  изоляции сбоев, что и `_supervise()` в Python-версии. Проверено вживую: рефреш токена
  из кэша, TLS-хендшейк, `JOIN` подтверждён Twitch-сервером на реальном канале.
- **Важно (безопасность)**: cpp-httplib по умолчанию НЕ проверяет TLS-сертификаты —
  все клиенты обязаны идти через `twitch::make_https_client()`
  (`core/include/twitch_auth.hpp`), который включает проверку и указывает на бандл
  `core/certs/cacert.pem` (Mozilla CA bundle, как у curl — на Windows у OpenSSL нет
  доступа к системному хранилищу сертификатов без него). Тот же файл используется в
  `asio::ssl` для IRC. Не создавай новый `httplib::Client` напрямую.
- **Settings-backend готов**: `core/include/runtime_settings.hpp` (`runtime_settings.json`,
  идентичен `softforstream/runtime_settings.py`), `core/include/moderation.hpp` (мьют-лист,
  подключён в колбэки Twitch/YouTube воркеров — замьюченные не долетают до оверлея),
  `core/include/chat_commands.hpp` (`chat_commands.json`, CRUD + cooldown). Полный REST
  в `OverlayServer`: `GET/POST /api/settings`, `GET /api/media/status`,
  `POST|DELETE /api/media/{kind}/{ext}`, `GET/POST /api/commands`,
  `DELETE /api/commands/{trigger}`, `POST /api/commands/{trigger}/toggle`,
  `GET /api/rvc/health|models` (пока стаб — RVC-адаптер не построен). Проверено вживую
  через curl. **Грабли Windows**: `crow::HTTPMethod::DELETE` не компилируется, если до
  этого где-то подключился `<windows.h>` (он приходит транзитивно через Asio) — макрос
  `DELETE` из winnt.h ломает enum-константу. Используй `crow::HTTPMethod::Delete`
  (Crow сам даёт этот безопасный алиас, см. `common.h`).
- Qt6.8.3 (msvc2022_64) прекомпилированный скачан в `qt/` через `aqtinstall` (dev-time
  only, см. `docs/BUILD.md`).
- **`gui/` переписан на QML (Qt Quick), Widgets-версия полностью удалена** — 6 страниц
  (`gui/qml/pages/*.qml`: Overlay/Tts/Alerts/Commands/Muted/Rvc), общая тема-синглтон
  `gui/qml/Theme.qml`, переиспользуемые компоненты в `gui/qml/components/`
  (`GlassCard` — настоящий блюр за карточкой через `ShaderEffectSource`+`MultiEffect`,
  `PillButton`, `NavButton`, `GlassTextField`, `GlassComboBox` — с собственной иконкой-
  шевроном вместо нативной стрелки, `GlassSlider`, `GlassToggle`, `MediaSlot`,
  `LinkChip`, `SectionHeader`). `gui/api_client.hpp/.cpp` — тот же REST-клиент, что был
  у Widgets-версии, адаптирован под JS-колбэки (`Q_INVOKABLE` + `QJSValue`), доступен в
  QML как context property `api`. Все 6 страниц реально дёргают `core`'s REST и были
  визуально проверены (см. ниже) — грузят/сохраняют настройки, алерты — загрузка файлов
  через `QtQuick.Dialogs.FileDialog`, команды — таблица через `Repeater`, RVC — статус
  пингуется таймером раз в 5 сек.
- **Полностью пройден живой цикл проверки на macOS через Homebrew Qt6** (`brew install
  qt cmake`, сборка через отдельный wrapper-CMakeLists вне репозитория, не по-настоящему
  `core` — тот Windows-only, а сама `gui/` кросс-платформенна). Рендер снимался в PNG
  через `QQuickWindow::grabWindow()` (не системный screen-capture, разрешений не
  требует). Найдены и исправлены две реальные проблемы по пути:
  - `QQuickStyle::setStyle("Basic")` обязателен **до** создания `QGuiApplication` —
    нативный стиль (macOS/Windows) запрещает кастомизацию `background`/`handle` у
    `Slider`/`ComboBox`/etc., тихо их игнорируя. Установлено в `main.cpp`.
  - `pragma Singleton` в `Theme.qml` сам по себе не регистрирует синглтон в сборке —
    без явного `set_source_files_properties(qml/Theme.qml PROPERTIES
    QT_QML_SINGLETON_TYPE true)` в CMake, Qt тихо (без единой ошибки в рантайме)
    резолвит `Theme` как обычный композитный тип без статических свойств — весь UI
    тогда рендерится с дефолтными цветами Qt Quick (белый фон, чёрные рамки, невидимый
    белый текст на белом фоне). Не путать с реальным багом — выглядит один в один как
    сломанная тема, а на деле чисто build-time нюанс.
  - `Qt6::QuickEffects` и `Qt6::QuickDialogs2` нужно явно перечислить в
    `find_package(Qt6 REQUIRED COMPONENTS ...)` и `target_link_libraries` — импорт
    `QtQuick.Effects`/`QtQuick.Dialogs` в самом `.qml` недостаточен.
  - Реальный блюр (`MultiEffect`) требует GPU-рендера сцен-графа: под
    `-platform offscreen` Qt Quick по умолчанию тихо падает на софтверный 2D-бэкенд
    (`qt.scenegraph.general: Loading backend software`), который не поддерживает
    шейдерные эффекты вообще — блюр рендерился пустым без единой ошибки. Форсирование
    `QT_QUICK_BACKEND=rhi` + `QSG_RHI_BACKEND=metal` вместе с offscreen-платформой на
    macOS приводило к сегфолту — рабочий вариант для проверки: обычное (не offscreen)
    окно, оно и с реальным GPU-рендером, и с чистым `grabWindow()` без плашки на экране.
  - Известный мелкий баг, не решён (не блокирует, echo в коде): `MultiEffect.maskEnabled
    + maskSource` красит карточку сплошным белым вместо скругления углов блюра —
    сейчас в `GlassCard.qml` блюр обрезается прямоугольно (без маски), скруглённая
    рамка/тинт поверх визуально прячут почти весь эффект, но по строгому пикселю угол
    блюра квадратный под скруглённым бордером. Чинить отдельно, если будет заметно
    глазом на реальной сборке.
- **Ещё не реализовано**: чат-команды не пишут ответы обратно в Twitch IRC (воркер
  read-only), Twitch EventSub (алерты), OBS auto-connect, TTS/RVC-адаптеры, трей,
  установщик.
